# instructions/00_overview.md

# 命令システム概要

## 目的

本書はゲーム内で使用するAIプログラムおよび命令システム全体の設計方針を定義する。

個々の命令の詳細仕様は各カテゴリの仕様書に記載し、本書では命令システム全体の共通仕様のみを扱う。

---

# 基本理念

本ゲームのAIは、プレイヤーが作成したプログラムを実行してロボットを自律制御する。

プログラムはリアルタイムで繰り返し実行され、現在の戦況に応じてロボットの行動を決定する。

プレイヤーは戦闘中に直接ロボットを操作することはできない。

---

# プログラムモデル

プログラムはフローチャートで構成される。

各命令はノードとして表現され、ノード同士を接続することで実行順序を定義する。

プログラムは開始ノードから実行され、接続に従って次の命令へ遷移する。

各Instruction Definitionは、命令が選択できる出力パスを定義する。

Nodeは出力パスIDごとの接続先Node IDを`connections`として保持する。命令実装は選択した出力パスに対応する接続先を解決し、命令実行結果を`nextNodeId`へ統一する。接続自体は条件を評価しない。

接続は制御フローだけを表し、値の受け渡しには使用しない。命令間の値の受け渡しにはレジスタまたはメモリを使用する。

---

# 実行モデル

AIはゲーム内の一定周期（Tick）ごとに実行される。

各Tickでは以下の順序で処理される。

1. センサー情報取得
2. プログラム実行開始
3. 命令実行
4. CPU制限到達または終了命令で停止
5. 行動結果をシミュレーターへ送信

プログラムは戦闘開始時に開始ノードから実行される。CPU不足でAI実行を終了した場合、次Tickは実行できなかったNodeから再開する。

レジスタ、フラグ、コールスタック、永続AIメモリはTickをまたいで保持される。

---

# 命令の構成

すべての命令は以下の要素で構成される。

* Instruction ID
* パラメータ定義
* 出力パス定義
* 実行時の副作用
* CPU消費量
* Implementation ID

命令の見た目や編集方法はエディタが担当し、命令システムは動作のみを定義する。

# 命令カテゴリ

命令は以下のカテゴリに分類する。

## 制御命令

プログラムの進行を制御する。

例

* 開始
* 終了
* 待機

---

## 分岐命令

条件に応じて実行経路を変更する。

例

* IF
* 比較
* ジャンプ
* CALL
* RETURN

---

## センサー命令

センサー情報を取得する。

例

* 敵の有無
* 距離
* 方位
* 被弾状態

---

## 演算命令

数値演算を行う。

例

* 加算
* 減算
* 乗算
* 除算
* 比較

---

## メモリ命令

内部状態を保存・取得する。

例

* 代入
* 読込
* 初期化

---

## 行動命令

ロボットの行動を指示する。

例

* 移動
* 旋回
* 武器使用
* 格闘
* 停止

## 特殊命令

その他の特殊処理を行う。

例

* 乱数取得
* デバッグ
* 将来追加命令

# 命令の実行

1つの命令は1回の実行で完結する。

命令はExecution Contextを参照し、必要に応じてExecution Context Changesを生成する。変更要求は命令の正常終了後にAI Engineがまとめて反映する。

命令自身は時間経過を持たない。

継続動作はAIが毎Tick繰り返し実行されることで実現する。

---

# CPU制限

1Tickで使用できるCPUコストには上限を設ける。

次の命令のCPUコストがCPU残量を超える場合、その命令を実行せず、そのTickのAI実行を終了する。

CPU上限によって終了した場合も、生成済みの行動要求はそのTickのシミュレーションで使用する。

CPU不足の場合、次のTickは実行できなかったNodeから再開する。命令実行結果の`interruptTick`が`true`の場合は、`nextNodeId`が示すNode、または`nextNodeId`が`null`ならProgramのStart Nodeから再開する。

条件分岐を含む循環は、Tick開始時に更新されるセンサー値やAI Runtime Stateの変化によって脱出できる可能性があるため、構造エラーとしない。

CPU上限はゲームバランスに応じてGame Rule Definitionで変更可能とする。

各命令のCPUコストはInstruction Definitionで定義する。0以上かつ1TickのCPU上限以下とし、0を許容する。

---

# レジスタとメモリ

AIは計算結果や状態保持のためにレジスタおよびメモリを利用できる。

レジスタ・メモリの個数と型はAI実行エンジンの仕様で定義する。どちらもAI Runtime Stateの一部としてTickをまたいで保持される。

命令はこれらを介して情報を受け渡す。

---

# データ型

命令で扱う値は共通のデータ型を使用する。

数値は共通データ規約に従い、符号付き32bit整数とする。小数は使用しない。数値型には汎用的な`integer`ではなく、`distance`、`degree`、`tick`、`cpu_cost`、`count`など値の意味を表す型を使用する。

数値以外には、真偽値、列挙値、レジスタ・フラグ・メモリ・Node・Master Dataへの参照型を使用できる。

新しい型を追加できる設計とする。

---

# エラー

命令そのものは構文エラーを持たない。

構造上の問題は Program Validator が検出する。

実行時エラーになった命令のCPU消費とExecution Context Changesは反映しない。そのTickのAI実行を終了し、次TickはProgramのStart Nodeから開始する。それ以前に正常終了した命令の変更と行動要求は維持する。

ゲーム全体を停止させる例外は発生させない。

---

# 拡張性

新しい命令は既存命令を変更せず追加できることを原則とする。

追加時に必要となる情報は以下とする。

* 命令ID
* 表示名
* カテゴリ
* パラメータ定義
* 出力パス定義
* CPU消費量
* 実行処理
* Implementation ID

既存カテゴリに属さない場合は、新カテゴリを追加できる。
