# 12_common_data_conventions.md

# 共通データ規約

## 目的

本書は、すべてのモジュールと保存データで使用する識別子、数値、列挙型、JSON、バージョンの共通規約を定義する。

---

# ID

IDは、スコープ内で一意なローカルIDと、保存データ間で一意なグローバルIDに分類する。IDは作成後に変更しない。

## ローカルID

ローカルIDの形式は以下とする。

```
{データ種別の識別子}_{データ種別ごとの連番}
```

例

* `node_1`
* `node_2`

ノードIDはProgram内で一意とする。異なるProgramに同一ノードIDが存在することを許容する。

## ローカルIDの発番

データ種別ごとに次回発番値を発番ソースとして保持し、対象データとともに保存する。

データ全体をコピーする場合は発番ソースもコピーし、コピー後はその発番ソースを使用する。

削除したIDは再利用せず、発番ソースの値を戻さない。

データの一部だけをコピー＆ペーストする場合は、ペースト先の発番ソースを使用してIDを新たに発番する。

発番値は1から開始する符号付き32bit整数とする。発番時に上限を超える場合はエラーとし、IDを発番しない。

## グローバルID

以下はUUIDを含むグローバルIDを持つ。

* Program：Editorで新規作成する際に発番する
* Robot設計データ：Editorで新規作成する際に発番する
* Replay Data：通常シミュレーション開始時に発番する
* Master Data：Master Dataの定義作成時に発番する

形式は以下とする。

```
program_{uuid}
robo_{uuid}
replay_{uuid}
```

Master Dataの接頭辞は`15_master_data.md`でデータ種別ごとに定義する。

Game Session内の実行時Robot IDは`robot_{Game Session内連番}`形式のローカルIDとする。参加者配列の先頭から`robot_1`、`robot_2`の順に発番し、Game Session内で一意とする。同じRobot設計データから生成した複数のRobotは、それぞれ異なる実行時Robot IDを持つ。

UUIDは識別のみに使用し、共通乱数生成器のシードやゲームロジックの分岐に使用しない。

ProgramまたはRobot設計データを別データとして複製する場合は新しいグローバルIDを発番する。同一データの編集ではIDを変更しない。

---

# 整数

ゲームロジックと保存データで扱う数値は符号付32bit整数とし、小数は使用しない。

例外として、擬似乱数生成器の内部演算と`nextUint32()`の戻り値に限り符号なし32bit整数を使用できる。JSONへの保存時は同じ32bitビット列を符号付き32bit整数として保存する。

```
-2147483648 <= value <= 2147483647
```

演算結果または入力値が範囲を超える場合はオーバーフローエラーとする。飽和、桁あふれによる巡回、暗黙の切り捨ては行わない。

オーバーフローでゲーム全体を停止させない。対象処理を失敗とし、代替値または代替動作は個別仕様で定義する。

JSON読込時も、整数であることと符号付32bit整数の範囲内であることを検証する。

---

# 角度

角度は度数法の整数で表現する。実行中は負の値および360以上の値を許容する。

JSONへの保存時は以下の範囲へ正規化する。

```
0 <= angle < 360
```

360度は0度と同値とする。

---

# Tick

Tickは符号付32bit整数とし、通常シミュレーション開始時を0とする。

通常シミュレーションが1回完了した場合のみTickを1増加する。入力待ち、一時停止、再描画、または通常シミュレーションが完了しなかった場合はTickを増加しない。

リプレイ再生はReplay Dataに記録されたTickを使用し、通常シミュレーションのTickを進めない。

Tickの増加によって符号付32bit整数の上限を超える場合はエラーとし、Tickを更新しない。

---

# 列挙型

数値で表現しない状態や種別は列挙型とする。列挙値は内容を端的に表す小文字の英単語とし、JSONでは文字列として保存する。

```json
{
  "status": "destroyed"
}
```

列挙型が取り得る値と未知の列挙値を受け取った場合の扱いは個別仕様で定義する。

---

# JSON保存形式

Program、Robot設計データ、Replay Dataなどの保存形式はJSONとする。JSONは人が直接編集することを前提とせず、内部表現と同じ構造とすることを許容する。

すべての保存JSONは以下の共通ヘッダを持つ。

```json
{
  "dataType": "program",
  "formatVersion": "0.1.1",
  "payload": {}
}
```

`dataType`は保存データの種別を表す文字列列挙値、`formatVersion`は互換性判定用のバージョン、`payload`はデータ種別ごとの本体とする。

JSONオブジェクトのプロパティ順序は意味を持たない。配列の要素順序は、個別仕様で順序不問と明記しない限り意味を持つ。

---

# JSON読込時の検証

保存JSONの読込時は、JSONスキーマによる構造検証に加えて、個別仕様で定義された意味的制約を検証する。

意味的制約には、該当する場合、IDの一意性、参照先の存在と種別、フィールド間の整合性、順序、およびイベントによる状態遷移を含む。対象データに適用する具体的な制約は個別仕様で定義する。

すべての検証が成功するまで、読込途中のデータを他モジュールへ公開しない。不正な値や欠損した参照を暗黙の既定値または別の参照先へ置換しない。

---

# バージョン

形式は以下とする。

```
{major_version}.{minor_version}.{build_version}
```

majorは0以上、minorとbuildは1以上の整数とし、初期バージョンは`0.1.1`とする。

プロジェクトオーナーから変更指示があるまでmajorは`0`に固定する。

* JSON構造を変更する場合も、majorの固定期間中はmajorを変更しない。変更後のminorとbuildは、構造変更前にプロジェクトオーナーが決定する
* JSON構造に変更がなく仕様、軽微なロジック、またはMaster Dataを変更した場合はminorを1増加し、buildを1に戻す。これらの変更によってシミュレーション結果が変わることを許容する
* 仕様に変更がないコード修正ではbuildを1増加する
* majorはプロジェクトオーナーの明示的な指示がある場合のみ変更する。その場合はminorとbuildを1に戻す

---

# 後方互換性

読込時は`dataType`と`formatVersion`を検証する。

* 古いmajorから変換可能な場合は現在の形式へマイグレーションする
* 未対応のmajorは安全に読込を拒否する
* 現在より新しいmajorは安全に読込を拒否する
* マイグレーションは元データを変更せず、新しいデータとして生成する

マイグレーション後のデータも現在のスキーマで検証する。
