# 14_determinism_rules.md

# 決定論規則

## 目的

本書は、実行端末やブラウザに依存しないシミュレーション結果を得るための共通規則を定義する。

---

# 同一性の保証範囲

以下が同一であれば、実行端末にかかわらず同一のWorld State変化とシミュレーション結果を得る。

* 完全なアプリケーションバージョン
* Program
* Robot設計データとパーツ構成
* Master Data
* マップと初期配置
* ゲームルール
* 初期World State
* 初期乱数シード

アプリケーションバージョンまたはMaster Dataが異なる場合は、majorバージョンが同じでも異なるシミュレーション過程と結果を許容する。

---

# 整数演算

ゲームロジックの数値演算は符号付き32bit整数で行う。浮動小数点数をゲームロジックの結果として保持しない。

各基本演算の結果ごとに符号付き32bit整数の範囲を検証する。演算結果が範囲外となる場合はオーバーフローエラーとする。飽和、巡回、暗黙の切り捨ては行わない。

複数の演算からなる処理は、個別仕様で定めた順序で評価する。実装都合による演算順序の入れ替えを行わない。

---

# 決定論的な文字列順序

個別仕様でIDの文字列昇順を使用する場合は、ASCIIコード順で比較する。IDは共通データ規約および個別仕様で定義されたASCII文字だけで構成する。

実行環境、言語設定、地域設定によって結果が変わるロケール依存の文字列比較を、保存順序、処理順序、または同順時の優先順位の決定に使用しない。

---

# 丸め

整数の除算で端数が生じた場合は、除算ごとに即座に四捨五入して符号付き32bit整数へ確定する。

端数がちょうど半分の場合は0から遠い整数へ丸める。

* `1.5` は `2`
* `-1.5` は `-2`

除数が0の場合は0除算エラーとする。エラー時の代替動作は個別仕様で定義する。

---

# 乱数生成器

Game Sessionは擬似乱数生成器を1つ持つ。すべてのゲームロジックはこの乱数生成器を共有する。

乱数生成器のアルゴリズムは`xorshift32`とする。内部演算に限り符号なし32bit整数を使用できる。

## 初期化

初期シードは符号付き32bit整数とする。シードの32bitビット列を符号なし32bit整数として解釈し、初期内部状態とする。

初期内部状態が0の場合は`0x6D2B79F5`へ置き換える。

開発言語または実行環境の乱数生成器を使用できるのは、Game Session開始前の初期シード生成時のみとする。Game Session開始後は使用しない。

初期シードはGame Sessionとリプレイ保存データへ記録する。

## 内部状態の更新

符号なし32bit整数`x`に対し、以下の順序で演算する。すべてのシフト、排他的論理和、切り捨ては32bitで行う。

```
x = x XOR (x << 13)
x = x XOR (x >>> 17)
x = x XOR (x << 5)
```

更新後の`x`を新しい内部状態とし、同じ値を`nextUint32()`の戻り値とする。

乱数生成器の内部状態はWorld Stateが保持し、Simulatorだけが更新を確定できる。JSONでは同じ32bitビット列を符号付き32bit整数として保存する。

## 乱数API

`nextUint32()`は乱数内部でのみ使用する。戻り値は0以上`4294967295`以下とする。

`nextInt(minInclusive, maxExclusive)`は以下の範囲の符号付き32bit整数を返す。

```
minInclusive <= result < maxExclusive
```

範囲変換は以下とする。

```
range = maxExclusive - minInclusive
result = minInclusive + (nextUint32() % range)
```

`range`は乱数の範囲変換中に限り符号なし32bit整数として扱う。

`minInclusive`が`maxExclusive`以上の場合はエラーとし、乱数内部状態を進めない。

剣余による範囲変換のごく小さな偏りは仕様として許容する。

# 乱数消費順

Game SessionはRobotの参加者配列を保持する。AIは参加者配列の先頭から順に実行する。

各Robotの行動要求はAI実行順にかかわらず、そのTick内で同時に行われたものとしてSimulatorが解決する。

乱数はゲームループと各個別仕様で定めた処理順で消費する。非同期処理、並列処理、ワーカーの完了順を乱数消費順の決定に使用しない。

AI Engineが乱数命令を実行する場合、Simulatorは現在の乱数内部状態をExecution Inputへ格納する。AI EngineはExecution Context内の作業コピーのみを更新し、Execution Resultに更新後の乱数内部状態を格納する。Simulatorは次のRobotのAIを実行する前にWorld Stateへ反映する。

---

# Replay Data

リプレイ再生時は乱数生成器を実行しない。乱数によって発生した結果はWorld Stateの変更内容としてReplay Dataへ記録する。

読込可能なReplay Dataは記録時と同じ過程と結果を再生する。記録された`masterDataVersion`が現在のData Repositoryと一致しない場合は、Replay Dataの読込を安全に拒否する。

Replay Dataは初期World StateとTickごとの変更イベント配列を保持する。同一Tick内の配列順を適用順序とする。変更イベントは更新後Robot状態、弾の生成、更新、削除、ゲーム進行状態と勝敗結果、Random Stateの更新を表す。

---

# バージョン

初期公開に向けた開発期間中は`formatVersion`と`masterDataVersion`を`0.1.1`に固定する。この期間中のMaster Data、JSON構造、仕様またはロジックの変更によって、シミュレーション過程と結果が変わることを許容する。

プロジェクトオーナーが初期公開とバージョン更新の開始を明示した後は、Master Dataの追加、削除、設定値変更と軽微なロジック変更によってminorバージョンを更新する。これらの変更によってシミュレーション結果が変わることを許容する。

JSON構造など互換性を壊す変更でも、majorの固定期間中はmajorを変更しない。majorはプロジェクトオーナーの明示的な指示がある場合のみ変更する。
