# instructions/concept.md

# AIプログラミング共通概念

## 目的

本書はAIプログラム全体で共通して使用する概念および用語を定義する。

各命令仕様書は本書の定義を前提とし、同一内容を重複して記述しない。

本書では実装方法ではなく、概念のみを定義する。

---

# AI

AIとは、ロボットを制御するプログラム全体を指す。

AIはプレイヤーが作成する。

戦闘中にAIは一定周期で繰り返し実行される。

AI自身は画面表示や物理演算を行わない。

---

# プログラム

プログラムとは、命令ノードを接続したフローチャート全体を指す。

プログラムは開始ノードを起点として実行される。

実行順序はノード間の接続によって決定される。

---

# ノード

ノードとは、プログラムを構成する最小単位である。

1つのノードは1つの命令を表す。

ノードは以下の情報を持つ。

- Node ID
- Instruction ID
- Parameter Values
- 出力パスごとの接続先Node ID

ノードはプログラム編集時のみ変更できる。

---

# 接続

接続とは、ノード間の実行順序を定義するリンクである。

Nodeは、Instruction Definitionが定義した出力パスIDをキー、接続先Node IDを値とする`connections`を保持する。

Instructionは条件判定結果に応じて出力パスIDを選択する。接続自体は条件を評価せず、選択された出力パスに対応する接続先が次に実行されるNodeとなる。

接続は制御フローだけを表し、値の受け渡しには使用しない。

---

# Tick

Tickとはゲーム内部時間の最小更新単位である。

ゲーム内のすべての処理はTick単位で進行する。

AIもTickごとに実行される。

Tickの長さはゲーム全体で共通とする。

---

# AI実行

AI実行とは、1Tick内でプログラムを評価する処理を指す。

AI実行では以下を行う。

- センサー情報取得
- 命令実行
- 行動決定

AI実行の結果はシミュレーターへ渡される。

---

# CPU

CPUとは、1Tick中に実行できる命令量を表す論理的な制限である。

CPUは現実のCPU性能とは無関係であり、ゲームバランスのための概念である。

各命令はCPUを消費する。

CPU上限に到達すると、そのTickのAI実行は終了する。

CPU量や消費値はAI実行エンジンで定義する。

---

# 命令

命令とは、AIが実行する処理単位である。

命令はExecution Contextを読み取り、`nextNodeId`、Execution Context Changes、`interruptTick`を含む命令実行結果を生成する。変更要求は命令の正常終了後にAI Engineが反映する。

命令はカテゴリごとに分類される。

Instruction Definitionと命令実行契約の詳細は`docs/specs/current/instructions/instruction_model.md`で定義する。

---

# パラメータ

パラメータとは、命令ごとに保持する設定値である。

例

- 定数
- 比較演算子
- 対象レジスタ
- 武器番号

パラメータはプログラム編集時に設定し、戦闘中には変更されない（命令仕様で明示される場合を除く）。

---

# レジスタ

レジスタとは、AIが計算結果を保持するための作業領域である。

主な用途は以下とする。

- センサー値の保持
- 演算結果の保持
- 命令間の値受け渡し

レジスタ構成はAI実行エンジンが定義する。

レジスタはAI Runtime Stateの一部としてTickをまたいで保持される。

---

# メモリ

メモリとは、Tickをまたいで保持されるAI内部状態である。

主な用途は以下とする。

- 状態管理
- カウンタ
- フラグ
- タイマー

メモリとレジスタはどちらもTickをまたぐ。個数、型、アクセス方法の違いは個別仕様で定義する。

---

# フラグ

フラグとは、真偽値によって状態を表現するデータである。

命令の結果やAI内部状態を保持する用途で使用する。

実装方法はAI実行エンジンで定義する。

フラグはAI Runtime Stateの一部としてTickをまたいで保持される。

---

# センサー情報

センサー情報とは、ロボットが外界から取得できる情報である。

例

- 敵の存在
- 距離
- 方位
- 被弾
- 障害物

AIはセンサー情報のみを通して外界を認識する。

---

# 行動命令

行動命令とは、ロボットへ行動要求を送る命令である。

例

- 移動
- 横移動
- 旋回
- 発射
- 格闘
- 停止

行動命令は即座に物理演算を行わず、シミュレーターへ要求を送る。

---

# 行動要求

行動要求とは、AIがシミュレーターへ送信する指示である。

行動要求は移動系、戦闘系の2カテゴリに分ける。同一Tick内で同じカテゴリの行動要求が複数回正常に生成された場合は、そのカテゴリで最後に生成された要求だけを保持する。異なるカテゴリの要求は同一Tickで併用でき、カテゴリ間では競合しない。

- 移動系：前進、後退、左旋回、右旋回、左横移動、右横移動、停止
- 戦闘系：武器切替、武器発射、格闘

Action Requestsは`movement`、`combat`を持ち、要求がないカテゴリは`null`とする。武器切替要求は切替先の右手または左手を`hand`として持つ。SimulatorがRobot Body Definitionの`weaponMount`を使用してSlot IDへ解決する。

各行動要求は`type`を判別子とする。`type`は何をするかを表し、そのほかのプロパティは、どのように実行するか、および何をもって行動完了とするかを行動ごとに定義する。すべての行動要求を同じプロパティ構造にはせず、`type`ごとの判別可能な共用体として扱う。

AI EngineはRobotの現在の動作段階を考慮せず、Programの実行結果としてそのTickの行動要求を生成する。AI Engineは行動要求を生成するまでを担当し、その要求がSimulatorに採用されたか、または実際に完了したかを判定しない。

実行待機命令に限り、Execution Inputが公開するカテゴリ別の`idle`または`running`だけを参照できる。現在の動作段階、進捗、採用した要求、次動作の内容は参照しない。

SimulatorはRobotの現在の動作状態と受け取った行動要求を照合し、要求を採用、上書き、キャンセル、次動作として保持、または無視する。採用した行動要求を実際の動作へ変換し、完了条件を判定する責務はSimulatorが持つ。要求が採用または実行されないことはAI命令の実行時エラーではない。

Simulatorが採用した行動要求は、完了またはキャンセルされるまでTickをまたいで有効とする。後続TickのAction Requestsで同じカテゴリが`null`であっても、現在の行動または保持中の次動作を取り消さない。

Robotの行動は予備動作、実動作、事後動作の順に進行する。予備動作はキャンセル可能とし、キャンセルした場合は事後動作を行わず、新しい行動の予備動作へ移る。実動作をキャンセルできるかは行動ごとの個別仕様で定義する。実動作を開始した行動は、正常完了またはキャンセルのどちらであっても、次の行動へ移る前に事後動作を完了しなければならない。事後動作はキャンセルできない。

各カテゴリは現在の行動とは別に次動作を最大1件保持する。次動作が開始する前に同じカテゴリの新しい要求を保持した場合は、最新の要求で次動作を上書きする。

現在の行動と新しい要求について、何をするか、およびどのように実行するかが同一であるかの判定項目は行動ごとの個別仕様で定義する。同一の場合、予備動作中または実動作中の新しい要求は無視し、現在の行動の完了条件を変更しない。異なる場合は、現在の動作段階に従って次のように処理する。

- 予備動作中：現在の行動をキャンセルし、新しい行動の予備動作へ移る
- キャンセル可能な実動作中：現在の行動をキャンセルして事後動作へ移し、新しい要求を次動作として保持する
- キャンセル不可能な実動作中：現在の行動を継続し、新しい要求を次動作として保持する
- 事後動作中：新しい要求を次動作として保持する

事後動作中は、現在の行動と同一の要求も新しい行動の要求として次動作へ保持する。現在の行動の事後動作が完了した後、保持した次動作の予備動作へ移る。

各段階の長さ、実際の効果を発生させる時点、実動作のキャンセル可否、および完了条件は、行動とSimulatorの個別仕様で定義する。

横移動はRobotの向きに対して左または右へ移動する要求である。格闘は前方の一定距離内にいる敵Robotへ急速に前進しながら打撃を加える要求である。停止は移動と旋回を中止する要求である。

武器発射要求は現在選択中のWeaponだけを対象とし、複数Weaponの同時発射を許可しない。ロックオン要求は定義しない。

シミュレーターは受け取った要求を次の更新処理で実行する。

AIはロボットの位置や速度を直接変更しない。

---

# ワールド状態

ワールド状態とは、ゲーム世界全体の現在状態である。

以下を含む。

- ロボット
- 弾
- エフェクト
- マップ
- 時間

AIはワールド状態を直接変更してはならない。

---

# Deterministic（決定論）

ゲームは決定論的シミュレーションを基本とする。

同じ完全なアプリケーションバージョン、Master Data、初期状態、入力、乱数シードで実行した場合、常に同一結果を得られることを必須とする。

これによりリプレイ、デバッグ、AI比較を容易にする。

---

# エラー

AI実行中の異常入力によってゲーム全体を停止させてはならない。

実行時エラーになった命令のCPU消費とExecution Context Changesは反映しない。そのTickのAI実行を終了し、次TickはProgramのStart Nodeから開始する。それ以前に正常終了した命令の変更と行動要求は維持する。

構文エラーや接続エラーは Program Validator が事前に検出する。

---

# Execution Context

## 概要

Execution Context（実行コンテキスト）とは、1TickのAI実行中のみ存在する実行環境である。

AIが命令を実行する際に必要な情報を一元管理し、命令間で共有する。

Execution Contextはゲーム世界そのものではなく、AIがゲーム世界を解釈・操作するための実行環境である。

AIのすべての命令はExecution Contextから情報を取得し、変更要求をExecution Context Changesとして返す。AI Engineが正常終了した命令の変更要求をExecution Contextへ反映する。

---

# 目的

Execution Context を導入する目的は以下とする。

- 命令の責務を統一する
- AI Engine と Simulator を分離する
- 命令が World State を直接変更しないようにする
- テストしやすい構造を実現する
- 将来の命令追加を容易にする

---

# ライフサイクル

Execution Context は AI実行開始時に生成される。

AI実行終了時に破棄される。

次のTickでは新しい Execution Context が生成される。

Execution Context自体はTickをまたがない。ただし、Execution Context内で更新したAI Runtime Stateと乱数内部状態の作業コピーはExecution Resultに含め、SimulatorがWorld Stateへ反映する。

---

# Execution Context が保持する情報

Execution Context は以下の情報を保持する。

## 実行情報

- Tick番号
- AI実行回数
- Program Counter
- CPU使用量
- CPU残量

---

## 実行対象

- Tick開始時のRobot状態スナップショット
- 実行中プログラム

---

## AI状態

- AI Runtime Stateから取得したレジスタの作業コピー
- AI Runtime Stateから取得したフラグの作業コピー
- 一時変数
- AI Runtime Stateから取得したコールスタックの作業コピー
- ジャンプ情報
- 永続AIメモリの作業コピー
- 乱数内部状態の作業コピー

一時変数が保持できる値は、符号付き32bit整数、boolean、文字列とする。一時変数はExecution Contextの破棄時に失われ、Tickをまたがない。

---

## センサー情報

AI開始時点で取得したセンサー情報を保持する。

例

- 敵情報
- 検出した弾の情報
- 距離
- 方位
- 被弾状態
- 自機状態

Robot検出情報は実行時Robot ID、相対位置、距離、方位、状態を持つ。弾検出情報はBullet ID、発射元Robot ID、相対位置、進行Vector、距離、方位を持つ。自機情報はRobot状態の読み取り専用スナップショットとする。

AI実行中はこのスナップショットを参照する。

同一Tick中にセンサー値が変化することはない。

---

## 行動要求

AIが決定した行動はExecution Contextへ蓄積する。

例

- 移動要求
- 横移動要求
- 旋回要求
- 武器発射要求
- 格闘要求
- 停止要求

命令は直接ロボットを動かさない。

AI実行エンジンはAI実行終了時に、行動要求、更新後のAI Runtime State、更新後の乱数内部状態をExecution ResultとしてSimulatorへ返す。

---

# World Stateとの関係

Execution Context はSimulatorがWorld Stateから生成したExecution Inputを読み取ることができる。Execution InputはRobot状態、AI Runtime State、センサー情報、Robotを中心とした座標系に変換した情報、乱数内部状態を含む。

スナップショットは読み取り専用とし、Execution Contextは World State を直接参照または変更しない。

ゲーム世界を変更する必要がある場合は行動要求を生成する。

```
World State

↓

Execution Input

↓

Execution Context

↓

Execution Result

↓

Simulator

↓

World State
```

---

# 命令との関係

すべての命令は

入力

↓

Execution Context

↓

処理

↓

Execution Context更新

という共通モデルで実行される。

命令は

- World State
- Robot
- Simulator

を直接操作しない。

---

# センサー取得

センサーはAI開始時に一括取得する。

Execution Input はその結果を保持し、Execution ContextはExecution Inputを介して参照する。

同一Tickではすべての命令が同じセンサー情報を参照する。

これにより実行順序による差異を防ぐ。

---

# 行動要求

行動命令はロボットを直接操作しない。

Execution Context に要求を書き込む。

AI終了後、AI実行エンジンが要求をExecution ResultとしてSimulatorへ返し、Simulatorが実行する。

例

```
MOVE命令

↓

Execution Context

moveForward = true

↓

Execution Result

↓

Simulator

↓

移動処理
```

---

# レジスタとメモリ

Execution Context はAI Runtime Stateに含まれるレジスタ、フラグ、コールスタック、永続AIメモリの作業コピーへアクセスできる。

AI実行中の更新は作業コピーに対して行い、AI実行終了後にSimulatorがExecution ResultのAI Runtime StateをWorld Stateへ反映する。

役割は以下の通りとする。

| 種類           | 寿命         |
| -------------- | ------------ |
| レジスタ       | Tickをまたぐ |
| フラグ         | Tickをまたぐ |
| コールスタック | Tickをまたぐ |
| 永続AIメモリ   | Tickをまたぐ |
| World State    | ゲーム全体   |

---

# CPU管理

CPU使用量はExecution Contextが管理する。

命令実行前にCPU残量を確認する。CPUコストが不足する場合は命令を実行せず、そのTickのAI実行を終了する。

命令が正常終了した場合はCPUコストを加算する。実行時エラーの場合は、その命令のCPU消費を反映しない。

CPU不足で終了した時点の実行できなかったNode、レジスタ、フラグ、コールスタック、永続AIメモリをAI Runtime StateとしてExecution Resultへ格納する。

CPU使用量とCPU残量はTickをまたがない。

---

# デバッグ

Execution Context はデバッグ情報の取得元となる。

例

- 現在命令
- 実行履歴
- CPU使用量
- レジスタ状態
- メモリ状態
- センサー値
- 行動要求

リプレイやデバッグ画面はExecution Contextを参照して表示を行う。

## Execution Context

Execution Context は将来的に以下の情報を保持できる。

- 命令プロファイラ
- 実行時間
- ブレークポイント
- トレースログ
- デバッグ変数
- AIバージョン情報

既存命令を変更せず拡張可能な構造とする。
